## Vision & Impact

Opérer la transition complète du Product Manager vers le **Product Builder**. Le rôle ne consiste plus à rédiger des tickets, mais à être un **navigateur d'incertitude** capable de choisir les bons modèles et de diriger des agents pour bâtir des solutions.

## Le saviez-vous ?

En 2026, la vision produit restera l'élément différenciateur majeur, car l'IA pourra tout produire, mais pas tout **concevoir**.

## Description

> *Devenez un Product Builder qui conçoit, prototype et livre.*

Écrire des specs pour que d'autres construisent ? C'est fini. Ce challenge vous fait devenir un Product Builder complet : vous concevez, vous prototypez, vous testez, vous itérez, vous livrez.

---

## Objectif du Challenge

**Ce que le participant doit produire :**

- [ ] Produit fonctionnel construit de A à Z
- [ ] Cycle complet : Discovery → Delivery
- [ ] Documentation du nouveau workflow PM

**Critères de réussite :**

- Produit utilisable par de vrais utilisateurs
- Cycle de < 1 semaine (vs semaines/mois avant)
- Autonomie sur toutes les étapes

## Pistes de Solution

### L'évolution du rôle PM

```
PM TRADITIONNEL (2020)           PRODUCT BUILDER (2026)
├── Écrire des specs        →   ├── Concevoir la solution
├── Prioriser le backlog    →   ├── Prototyper rapidement
├── Coordonner les équipes  →   ├── Tester avec users
├── Suivre les métriques    →   ├── Itérer en direct
└── Présenter les résultats →   └── Livrer le produit
```

### Les compétences du Product Builder

```
VISION (inchangé)
├── Comprendre le marché
├── Identifier les opportunités
├── Définir la direction
└── Prioriser impitoyablement

CONCEPTION (amplifié)
├── Décomposer l'intention
├── Designer les flows
├── Spécifier précisément
└── Anticiper les edge cases

CONSTRUCTION (nouveau)
├── Prompter efficacement
├── Utiliser les outils no/low-code
├── Piloter les agents de coding
├── Intégrer les APIs
└── Debugger les problèmes simples

VALIDATION (accéléré)
├── Tester avec vrais utilisateurs
├── Mesurer l'impact
├── Itérer en continu
└── Décider de pivoter/persévérer
```

### Workflow Product Builder

```
JOUR 1 : DISCOVERY
├── 2h : Recherche et interviews
├── 2h : Synthèse et hypothèses
├── 2h : Design de la solution (Figma ou Vibe Design)
└── 2h : Validation avec users potentiels

JOUR 2-3 : BUILD
├── 4h : Prototype fonctionnel (Claude Code / Cursor)
├── 4h : Intégrations (Auth, DB, APIs)
├── 4h : Polish et edge cases
└── 4h : Tests et corrections

JOUR 4 : LAUNCH
├── 2h : Déploiement
├── 2h : Documentation
├── 2h : Onboarding premiers users
└── 2h : Collecte feedback

JOUR 5 : ITERATE
├── Analyser les données
├── Prioriser les améliorations
├── Itérer sur le produit
└── Recommencer le cycle
```

### Exercice : Construire un produit

```
BRIEF :
Construire un produit complet qui résout un problème réel.

CONTRAINTES :
- Maximum 1 semaine
- Utilisable par au moins 5 personnes
- Valeur mesurable

LIVRABLES :
1. Produit fonctionnel déployé
2. 5+ utilisateurs actifs
3. Métriques de base en place
4. Feedback collecté
5. Prochaines itérations identifiées

ÉVALUATION :
- Le produit fonctionne-t-il ?
- Résout-il vraiment le problème ?
- Les utilisateurs l'utilisent-ils spontanément ?
- Quelle est la vélocité d'itération ?
```

### Outils du Product Builder

```
DISCOVERY
├── Interviews : Grain, Fireflies
├── Recherche : Perplexity, Claude
├── Synthèse : NotebookLM

DESIGN
├── Rapide : Claude Artifacts, v0
├── Détaillé : Figma
├── Vibe Design : Conversation Claude

BUILD
├── Code : Claude Code, Cursor
├── Backend : Supabase, Firebase
├── Intégrations : n8n, Zapier

MEASURE
├── Analytics : Plausible, Posthog
├── Feedback : Typeform, Canny
├── Metrics : Mixpanel, Amplitude
```
